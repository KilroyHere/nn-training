\section{Hypothesis}
\label{sec:hypothesis}
% ---------------------------------------------------------------
% State the question you want to investigate.
% What did you expect to find, and what did you actually find?
% ---------------------------------------------------------------

We investigate MPI acceleration of MLP training on MNIST through two families
of techniques: data parallelism (flat DP, hierarchical DP, and Local-SGD) and
model parallelism (sequential layer partitioning and pipelined variants). Our
hypothesis is that communication structure, not only rank count, determines
scaling: methods that reduce global synchronization pressure should achieve
better epoch-time scaling while retaining comparable optimization behavior.

The results support this framing. In data parallelism, Local-SGD provides the
best large-scale epoch-time reduction, flat DP remains the strongest fully
synchronous baseline, and hierarchical DP does not amortize its extra phases at
our current model size. In model parallelism, sequential partitioning remains
communication-bound, while pipeline performance is highly sensitive to
microbatch count and load balance.

\section{Prior Work}
\label{sec:prior}
% ---------------------------------------------------------------
% ~3 previous works this project builds on, is motivated by,
% or will be compared against.
% ---------------------------------------------------------------

Distributed training has been studied across both data-parallel and
model-parallel axes. \citet{ref1} established synchronous data-parallel
training via collective communication, which directly motivates our flat-DP
and hierarchical-DP baselines. \citet{ref2} showed that periodic weight
synchronization (local SGD) reduces communication overhead while retaining
convergence, motivating our \(K\)-step synchronization experiments.
For model parallelism, \citet{ref3} demonstrated that pipeline-aware
scheduling is essential and that bubble overhead is a first-order performance
factor; our implementation adopts the same interleaved forward-backward
schedule and uses the alpha-beta model to characterize when bubble reduction
is overwhelmed by accumulated MPI startup latency.

\section{Empirical Methodology}
\label{sec:methodology}
\subsection{Problem Setup}
We evaluate two distinct parallelization families on MNIST.

\noindent\textbf{Data-parallel family (Serial, Flat DP, Hierarchical DP, Local SGD).}
These experiments use the main scaling model
\(784 \rightarrow 256 \rightarrow 128 \rightarrow 64 \rightarrow 10\), trained
with SGD and cross-entropy under fixed optimizer settings and matched data
splits. The execution modes are:
\begin{itemize}[nosep]
  \item \textbf{Serial}: single-process baseline.
  \item \textbf{Flat MPI Data Parallel (DP)}: each rank computes local gradients
        and performs gradient all-reduce every step.
  \item \textbf{Hierarchical MPI DP}: per-step gradient synchronization is split
        into intra-node reduce, inter-node all-reduce (leaders), and intra-node
        broadcast.
  \item \textbf{Local SGD}: each rank performs local updates and averages model
        weights every \(K\) steps.
\end{itemize}
This family is used for the strong-scaling conclusions from \(1n4r\) to
\(4n128r\).

\noindent\textbf{Model-parallel family (MP and pipeline variants).}
These experiments use a deeper architecture with 8 weight matrices,
\(784\!\rightarrow\!512\!\rightarrow\!512\!\rightarrow\!256\!\rightarrow\!256\!\rightarrow\!128\!\rightarrow\!64\!\rightarrow\!32\!\rightarrow\!10\),
so layer partitioning across ranks is meaningful. Here we analyze sequential
model parallelism and pipeline behavior under batch-size, microbatch-count, and
world-size sweeps.

\subsection{Sequential Baseline}
\label{sec:baseline}
The sequential baseline uses the same MLP, loss, and optimizer as all MPI
variants. It runs a shuffled mini-batch loop over training data and reports
\texttt{epoch\_time\_ms} as \emph{training-loop-only} wall time (validation
evaluation is measured but excluded from epoch timing). Validation is excluded from epoch timing in all backends, so comparisons
reflect training cost only.

\subsection{Parallel Design}
\label{sec:parallel-design}
\subsubsection{Design Overview}
Serial training has no communication. Data-parallel methods synchronize gradients or weights with collectives. Model parallelism partitions layers and exchanges intermediate tensors point-to-point. For clarity, we therefore present separate algorithms for (1) serial training, (2) data-parallel training, and (3) model-parallel training.

\subsubsection{Serial Design}
\begin{algorithm}[H]
\caption{Serial training epoch}
\label{alg:serial}
\begin{algorithmic}[1]
  \Require shuffled indices $I$, batch size $B$, parameters $\theta$
  \Ensure updated $\theta$ after one epoch
  \For{$t = 0, B, 2B, \dots, |I|-B$}
    \State gather batch $(x, y)$ from $I$
    \State $g \leftarrow \nabla \ell(\theta; x, y)$
    \State $\theta \leftarrow \theta - \eta g$
  \EndFor
\end{algorithmic}
\end{algorithm}
\noindent\textbf{Serial.}
Single-process SGD over shuffled mini-batches; no MPI calls.

\subsubsection{Data-Parallel Designs}
\begin{algorithm}[H]
\caption{Data-parallel epoch (flat DP / hierarchical DP / local SGD)}
\label{alg:dp}
\begin{algorithmic}[1]
  \Require shuffled index list $I$, global batch size $B$, rank count $P$
  \Ensure updated model parameters after one epoch
  \For{global batch offset $t = 0, B, 2B, \dots, |I|-B$}
    \State gather local mini-batch $(x_r, y_r)$ from $I$
    \State compute local gradients $g_r \leftarrow \nabla \ell(\theta; x_r, y_r)$
    \If{mode is Flat DP}
      \State $g \leftarrow \texttt{AllreduceMean}(g_r)$
    \ElsIf{mode is Hierarchical DP}
      \State $g \leftarrow \texttt{IntraReduce} \rightarrow \texttt{InterAllreduce} \rightarrow \texttt{IntraBcast}$
    \ElsIf{mode is Local SGD}
      \State set $g \leftarrow g_r$; synchronize weights only every $K$ steps
    \EndIf
    \State apply SGD update with synchronized quantity (gradient or weight)
  \EndFor
\end{algorithmic}
\end{algorithm}
\noindent\textbf{Flat DP.}
Each rank processes a disjoint shard of each global data batch, computes local
gradients, then uses \texttt{MPI\_Allreduce} on layer gradients to form the
global mean gradient before applying the update. \\
\noindent\textbf{Hierarchical DP.}
Gradient synchronization is decomposed into three phases:
\texttt{MPI\_Reduce} within a node, \texttt{MPI\_Allreduce} across node
leaders, then \texttt{MPI\_Bcast} within each node. This reduces inter-node
participants but introduces extra synchronization phases. \\
\noindent\textbf{Local SGD.}
Ranks execute local updates independently and synchronize model weights via one
fused \texttt{MPI\_Allreduce} every $K$ steps (\(K \in \{10, 50\}\) in this
report). This trades statistical accuracy for lower communication frequency. \\

\subsubsection{Model-Parallel and Pipeline Designs}
\begin{algorithm}[H]
\caption{Model-Parallel batch step (Sequential / Pipeline / Load-Balanced)}
\label{alg:mp_unified}
\begin{algorithmic}[1]
  \Require ranks $P$, rank $r$, batch $(x,y)$, microbatches $M$ (if pipelined)
  \State \textbf{if} mode is Load-Balanced \textbf{then} assign $\theta_r$ by layer cost \textbf{else} assign naively
  \If{mode is Model Parallel}
    \State receive $a_{\text{in}}$ (or $x$), compute forward $a_{\text{out}}$, send to $r+1$
    \State receive $g_{\text{out}}$ (or $\nabla \ell$), compute backward $\nabla \theta_r$ and $g_{\text{in}}$, send $g_{\text{in}}$ to $r-1$
  \ElsIf{mode $\in$ \{Pipeline, Load-Balanced\}}
    \State split $(x,y)$ into $M$ microbatches; initialize $\nabla \theta_r^{\text{accum}} \leftarrow 0$
    \For{step $t = 0, \dots, M + P - 2$}
      \State \textbf{if} $t \in [0, M-1]$ \textbf{then} compute forward microbatch $t$, send $a_{\text{out}}$
      \State \textbf{if} $t - (P - 1) \in [0, M-1]$ \textbf{then} compute backward microbatch, add to $\nabla \theta_r^{\text{accum}}$
    \EndFor
    \State $\nabla \theta_r \leftarrow \nabla \theta_r^{\text{accum}} / M$
  \EndIf
  \State apply SGD update to $\theta_r$ using $\nabla \theta_r$
\end{algorithmic}
\end{algorithm}
\paragraph{Model Parallel and Pipelining.}
Layers are partitioned across ranks. Forward activations and backward gradients are exchanged via point-to-point MPI. We evaluate three partitioning and scheduling variants:
\begin{itemize}[nosep]
  \item \textbf{Model Parallel:} A non-pipelined schedule where a single global batch is passed sequentially across ranks, leaving $(P-1)$ ranks idle at any given time.
  \item \textbf{Pipeline Parallel:} The global batch is split into $M$ microbatches. Ranks execute a staggered schedule, interleaving forward and backward passes (offset by $P-1$ steps) to fill the pipeline. Gradients are accumulated and applied synchronously at the end.
  \item \textbf{Load-Balanced Pipeline:} Instead of a naive $L/P$ layer division, layers are assigned greedily based on computational cost (using $d_{\text{in}} \times d_{\text{out}}$ parameter counts) to equalize the workload per rank and minimize pipeline bottleneck stages.
\end{itemize}

\subsection{Communication Cost Model (Alpha-Beta)}
\label{sec:alpha-beta}
We use the standard latency-bandwidth model:
\begin{equation}
T_{\text{msg}}(m) = \alpha + \beta m,
\end{equation}
where \(\alpha\) is startup latency, \(\beta\) is per-byte (or per-float) transfer
cost, and \(m\) is message size.

\paragraph{Variable legend.}
\begin{itemize}[nosep]
  \item \(P\): total MPI ranks (world size); \(r\): rank index.
  \item \(N\): nodes, \(q\): ranks per node, with \(P = Nq\).
  \item \(\alpha\): per-message startup latency; \(\beta\): inverse bandwidth term.
  \item \(m\): generic message size; \(m_g\): gradient payload; \(m_w\): weight payload.
  \item \(m_a\): activation message size; \(m_b\): backward-gradient message size.
  \item \(T_{\text{msg}}(\cdot)\): one-message communication time.
  \item \(T_{\text{flat}}, T_{\text{hier}}, T_{\text{local,step}}, T_{\text{mp,step}}\): per-step costs for flat DP, hierarchical DP, Local-SGD, and model parallel.
  \item \(T_{\text{compute}}\): local compute time per step (excluding communication).
  \item \(T_{\text{sync}}(\cdot)\): synchronization cost term used in Local-SGD.
  \item \(K\): Local-SGD sync interval (average one sync every \(K\) steps).
  \item \(C_r\): compute time of stage/rank \(r\); \(C_0\): representative stage compute cost in the pipeline approximation.
  \item \(L\): pipeline depth (number of stages/ranks); \(M\): microbatches per global batch.
  \item \(\mathrm{bubble}(L,M)=\frac{L-1}{M+L-1}\): idealized pipeline bubble fraction.
\end{itemize}

\begin{center}
\footnotesize
\captionof{table}{ Alpha-Beta communication model summary by method.}
\label{tab:alpha-beta-summary}
\begin{tabular}{p{0.21\linewidth}p{0.74\linewidth}}
\toprule
\textbf{Method} & \textbf{Per-step or per-batch approximation} \\
\midrule
Flat DP &
\(T_{\text{flat}}(m_g) \approx 2(P-1)\alpha + 2\frac{P-1}{P}\beta m_g\). Scales well at moderate \(P\), but latency grows with rank count. \\
Hierarchical DP &
\(T_{\text{hier}}(m_g) \approx \left(2\log q\,\alpha + 2\frac{q-1}{q}\beta m_g\right) + \left(2(N-1)\alpha + 2\frac{N-1}{N}\beta m_g\right)\), with \(P=Nq\). Reduces inter-node participants, but adds extra sync phases. \\
Local SGD &
\(T_{\text{local,step}} \approx T_{\text{compute}} + \frac{1}{K}T_{\text{sync}}(m_w)\). Larger \(K\) lowers communication cost but can weaken statistical freshness. \\
Model Parallel &
\(T_{\text{mp,step}} \approx \sum_{r=1}^{P}\left(C_r + (\alpha+\beta m_a) + (\alpha+\beta m_b)\right)\). Stage imbalance and boundary traffic dominate runtime. \\
Pipeline Parallel &
\(T_{\text{batch}} = (M + L - 1)\max\!\left(\frac{C_0}{M}, \alpha + \frac{\beta m_0}{M}\right) + (L - 1)\min\!\left(\frac{C_0}{M}, \alpha + \frac{\beta m_0}{M}\right)\), \(\text{bubble}(L,M)=\frac{L-1}{M+L-1}\). More microbatches reduce bubbles, but too many can amplify latency cost. \\
\bottomrule
\end{tabular}
\end{center}

\subsection{Implementation Details}
\label{sec:impl}
The project is implemented in C++17 with MPI and BLAS-backed dense linear
algebra. To keep comparisons fair, serial and distributed backends share the
same model definition, optimizer semantics, data preprocessing, and epoch-timing
convention.

\paragraph{Shared runtime and kernels.}
All backends use BLAS-backed primitives for dense operations
(\texttt{sgemm/saxpy/sgemv}) instead of scalar loops. This reduces per-step
compute overhead and keeps arithmetic kernels consistent across experiments.

\paragraph{Backend-specific execution paths.}
The codebase isolates execution strategies into dedicated binaries and launch
scripts for serial, flat DP, hierarchical DP, local-SGD, and model parallelism.
This avoids mode-specific branching in hot loops and keeps experimental control
explicit at launch time.

\paragraph{Performance-focused optimizations.}
Training hot loops use buffer reuse and locality-aware data movement:
(i) preallocated mini-batch feature/label buffers, (ii) contiguous gather
writes, and (iii) reusable gradient/workspace buffers in the shared MLP path.
Release builds use \texttt{-O3}.

\paragraph{Fair-comparison controls.}
The serial path is intentionally tuned with the same kernel and build strategy
as parallel runs, so reported speedups are not inflated by avoidable single-rank
overhead. Since model parallelism uses a separate
forward/backward path, its performance depends less on the BLAS and
buffer optimizations above, and more on how evenly layers are distributed across ranks and how quickly activations and gradients can be passed between neighboring ranks.

noted otherwise, runs use 10 epochs and seed \(=42\).

\subsection{Results}
\label{sec:results}
DP and MP results are reported as two method families with different workload
goals and model shapes (Section~\ref{sec:setup}); comparisons are therefore
interpreted within family (DP vs DP, MP vs MP), not as direct absolute DP-vs-MP
rankings.

\subsubsection{Data-Parallel Results}
\paragraph{Epoch time vs.\ cluster size.}
Figure~\ref{fig:epoch-vs-cluster} shows epoch time vs cluster size with serial
baseline and ideal scaling reference. Local-SGD achieves the best time at
128 ranks (K=50 fastest), while flat DP remains more efficient than
hierarchical DP for this model size and workload regime. A likely explanation
is that \texttt{MPI\_Allreduce} in the flat-DP path benefits from mature,
topology-aware collective implementations in the system MPI stack, whereas the
explicit two-level hierarchical sequence (intra-reduce, inter-allreduce,
intra-bcast) adds extra synchronization phases that are not sufficiently
amortized at the current gradient payload.

\paragraph{Communication breakdown and synchronization cost.}
Timing breakdowns from logs show:
\begin{itemize}[nosep]
  \item flat DP: communication cost appears primarily in \texttt{allreduce\_ms},
        while \texttt{other\_ms} remains non-negligible at high ranks. Combined
        with better end-to-end epoch times, this is consistent with (but does
        not by itself prove) topology-aware optimization of
        \texttt{MPI\_Allreduce} by the MPI runtime; the exact per-call internal
        collective algorithm is not fully exposed in public documentation for
        the production MPI stack used on this system
        \citep{nersc-cray-mpich,nersc-perlmutter-network,hpe-intro-mpi}.
  \item hierarchical DP: cost is split into
        \texttt{intra\_reduce\_ms}, \texttt{inter\_allreduce\_ms}, and
        \texttt{intra\_bcast\_ms}, confirming additional synchronization phases.
  \item local-SGD: \texttt{sync\_ms} is much smaller than per-step DP comm cost
        because communication occurs only every \(K\) steps.
\end{itemize}
Accuracy trade-off is modest for the fastest local-SGD settings in these runs:
at 128 ranks, K=50 is fastest with a small validation-accuracy drop relative to
flat DP.

\paragraph{Local-SGD synchronization sensitivity (\(K\)).}
Across the tested Local-SGD settings (\(K\in\{10,50\}\)), increasing \(K\)
reduces synchronization overhead and improves epoch time in large-scale runs,
with only a modest validation-accuracy change in the current workload regime.

\subsubsection{Model-Parallel Results}

\paragraph{Serial vs.\ model parallel vs.\ pipeline (batch size sweep).}
Figure~\ref{fig:mp-batch} shows epoch time as batch size varies from 256
to 4096 at ws\,=\,8.
Sequential model parallel runs at $72$--$79$\,s/epoch regardless of batch
size — roughly $2\times$ the serial baseline of $36$--$39$\,s — because the
non-pipelined schedule serializes all ranks: total step time equals the
\emph{sum} of every rank's compute plus communication.
Pipeline parallel is strongly batch-sensitive: at batch\,=\,256 the
microbatch size is only 8 rows, making per-message latency the dominant
cost ($\approx\!140$\,s/epoch). As batch size grows and microbatches become
larger, pipeline time falls sharply, approaching the model-parallel line at
batch\,=\,2048 and crossing below it at batch\,=\,4096 when microbatch size
is large enough to amortize startup latency.

At $M\!=\!8$ (microbatch size 512 rows) pipeline epoch time is
$\approx\!39$\,s, nearly matching serial ($\approx\!37$\,s). At $M\!=\!32$
it rises to $\approx\!71.6$\,s, and at $M\!=\!64$ to $\approx\!81.2$\,s,

($90.1\%$).
The timing breakdown (Figure~\ref{fig:mp-mb}b) reveals why: forward
communication wait (\texttt{fwd\_comm}) — the time each rank sits idle
waiting for the next activation — grows from $\approx\!18$\,s at
$M\!=\!8$ to $\approx\!55$\,s at $M\!=\!64$, scaling roughly linearly with
$M$. Each of the $M$ microbatches incurs a separate MPI send latency
$\alpha$ regardless of message size; for small MNIST activations $\alpha$
dominates over bandwidth cost $\beta m$, so total idle time accumulates as
$M\alpha$ and overwhelms the theoretical bubble reduction.

\paragraph{Pipeline vs.\ load-balanced pipeline.}
Figure~\ref{fig:mp-ws} shows epoch time across world sizes ws\,$\in\!\{2,4,6,8\}$
at batch\,=\,4096 and $M\!=\!32$. Model Parallel is flat at $\approx\!72$\,s
regardless of world size, as expected. Pipeline improves modestly with world
size; the load-balanced variant reduces epoch time by $2$--$6\%$ over naive
round-robin assignment at ws\,=\,2,\,4,\,6 (Figure~\ref{fig:mp-bal}). The improvement peaks at ws\,=\,6 ($+6.1\%$),
where naive assignment places the two heaviest layers on the earliest ranks;
the balanced partitioner redistributes by cost. At ws\,=\,8 both strategies
assign exactly one layer per rank, so the result is identical ($+0.0\%$).
Across all configurations, serial ($\approx\!37$\,s) remains faster than
every parallel variant.

\section{Challenges and Obstacles}
\label{sec:challenges}
% ---------------------------------------------------------------
% Obstacles encountered.
% Hypotheses for things that did not work as expected.
% ---------------------------------------------------------------

\paragraph{Challenge 1: Microbatch count unexpectedly dominated pipeline performance.}
The bubble fraction $\frac{L-1}{M+L-1}$ decreases as $M$ grows, predicting
that more microbatches should reduce pipeline idle time and improve
throughput. Instead, $M\!=\!8$ nearly matched serial ($\approx\!39$\,s)
while $M\!=\!64$ was the slowest ($\approx\!81$\,s) despite having the
highest bubble fraction of $47\%$ vs $9.9\%$ respectively. The timing
breakdown revealed that forward communication wait (\texttt{fwd\_comm})
grows roughly as $M\alpha$: each microbatch incurs a fixed MPI startup
cost $\alpha$ regardless of message size, and for small MNIST activations
this latency dominates over bandwidth cost $\beta m$. The bubble model is
therefore incomplete — it only captures compute idleness and ignores
accumulated communication latency, which becomes the binding constraint
at large $M$.

\paragraph{Challenge 2: Flat \texttt{MPI\_Allreduce} outperformed our hierarchical design at scale.}
We expected hierarchical DP to win at higher rank counts by reducing
inter-node participants, but on Perlmutter the flat DP path was consistently
faster in end-to-end epoch time. A key obstacle was that this contradicted our
initial communication-structure intuition and forced us to revisit system-level
effects. Our working explanation is that the production MPI stack on
Perlmutter provides highly optimized, topology-aware collective algorithms for
\texttt{MPI\_Allreduce}, while our explicit two-level hierarchy
(\texttt{Reduce} $\rightarrow$ leader \texttt{Allreduce} $\rightarrow$
\texttt{Bcast}) introduces extra synchronization phases and software overhead
that were not amortized at our model size.

\paragraph{Challenge 3: Keeping cross-backend comparisons fair required strict measurement discipline.}
Serial, data-parallel, and model-parallel paths use different communication
patterns and partially different kernels, so naive timing comparisons were easy
to misinterpret. We had to enforce matched launch semantics, a shared epoch-time
definition, and an optimized serial baseline (BLAS-backed kernels, reuse-oriented
buffers, release builds) before interpreting speedups. This added engineering
overhead but was necessary to attribute differences to algorithmic choices
rather than mismatched implementation quality.
